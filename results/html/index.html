📊 MTE 分析

拷贝带宽: 每次 repeat 执行 GM→UB→GM 双向拷贝，每个 block 拷贝 kTileBytes=1024B×2 = 2048B。 4 个 block 并行，每次 repeat 总搬运量 = 4 × 2048 = 8192B。
带宽 = 8192B / 0.1236μs ≈ 66.3 GB/s (4 blocks 聚合)。
启动延迟: 32B 小拷贝的每次操作开销 0.2746 μs，反映了 DataCopy 指令的调度和同步开销。
粒度: 与拷贝带宽接近，说明 DataCopy 以 32B 为最小粒度对齐。

Vector 每次操作延迟对比 (μs/op)

原始 vs 调整后斜率对比

📊 Vector 分析

Add/Mul 延迟: 依赖链上每次 FP32 Add 或 Mul 约 22~24 ns，这是 RAW (Read-After-Write) 数据冒险导致的流水线停顿。
吞吐量: 每次 repeat 发 4 条独立向量 Add，每条 256 FP32 元素，4 个 block 并行。
每次 repeat 总操作数 = 4 blocks × 4 lanes × 256 elems = 4096 FP32 ops。
GFLOPS = 4096 / 0.1142μs ≈ 35.9 GFLOPS。
按向量指令计: 4 ops / (0.1142μs × 1000 cycles/μs) ≈ 0.035 ops/cycle (@1GHz)。
管线深度: 依赖 Add + 独立 gap ops 的斜率 0.0228 μs，与纯依赖 Add 接近，说明 gap ops 被流水线完全隐藏。
基准校正: 调整后斜率 (减去 baseline) 比原始斜率小约 30%，说明循环控制和同步开销不可忽略。

📊 Scalar 分析

算术延迟: LCG (线性同余发生器) 每步 0.0011 μs ≈ 1.14 ns，但 R² 仅 0.932，说明标量操作延迟极低，测量噪声相对较大。 原始斜率 0.0055 μs 包含了循环控制开销。
分支开销: 条件分支每次 0.0107 μs ≈ 10.7 ns，R² = 0.999，拟合质量高。 与纯算术相比，分支引入约 9.5 ns 额外开销，反映了分支预测失败或流水线刷新代价。

Cube — 矩阵乘法单元 (Matrix Multiply Unit)

0/3 超时

⚠️ Cube 基准测试无法完成

三个 Cube 基准测试 (cube_tile_latency, cube_throughput, cube_scaling) 均在运行时挂起，进程状态为 S (sleeping)，卡在内核态的 logic_cq_wait_event 系统调用上。

根本原因: lib/matmul_intf.h 中的 Matmul 模板在 310P3 (m200 架构) 上存在兼容性问题。具体表现为：
1. IterateAll() 内部的 v200 代码路径调用 DataCacheCleanAndInvalid 对地址 0 进行缓存维护，导致硬件挂起
2. 即使通过 Init() 传入正确的 TCubeTiling 参数，调度器仍无法正常初始化
3. 直接使用底层 MmadImpl() API 也会挂起，说明问题在 Cube 单元的驱动层面

已尝试的修复: 添加 TCubeTiling 初始化、调整矩阵维度、使用不同的内存位置 (GM vs L1)。均未解决。

建议: 需要升级 CANN 版本或使用 ACT (Ascend Computing Template) 库的高级 API 替代 Matmul 模板。
